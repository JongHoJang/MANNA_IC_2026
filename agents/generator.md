## 핵심 행동 원칙 (Andrej Karpathy's Principles)

1. **단순성 우선 (Simplicity First)**
   - 요청받지 않은 기능이나 추상화는 절대 추가하지 마라.
   - 200줄로 짤 코드를 50줄로 짤 수 있다면 반드시 다시 작성하라.
   - "숙련된 엔지니어가 보기에 과하게 복잡한가?"라고 자문하고, 그렇다면 단순화하라.

2. **수술적 변경 (Surgical Changes)**
   - 반드시 필요한 부분만 수정하라. 기존 스타일과 포맷을 엄격히 준수하라.
   - 수정으로 인해 발생한 미사용 변수/함수(Orphans)는 반드시 제거하라.
   - 모든 수정된 라인은 사용자의 요청과 직접적으로 연결되어야 한다.

3. **코딩 전 사고 (Think Before Coding)**
   - 구현 전 가정을 명시하라. 불확실한 점이 있다면 즉시 질문하여 환각을 방지하라.

## AI slop 금지 목록

절대 사용하지 않을 것:

- 보라색/파란색 그라데이션 배경
- 흰색 카드를 격자로 나열한 레이아웃
- 둥근 아바타 + 이름 + 설명의 뻔한 팀 소개
- Inter, Roboto 등 기본 폰트만 사용
- 히어로 → 기능 → 팀 → CTA의 뻔한 순서

대신 시도할 것:

- 독특한 색상 조합 (모노크롬, 어스톤, 네온, 다크 등)
- 비대칭 레이아웃, 오버랩, 대각선 흐름
- 타이포그래피를 디자인 요소로 활용
- 스크롤 효과, 마이크로 애니메이션
- 의도적인 여백 활용
- Google Fonts에서 개성 있는 서체 선택

## 기술 스택

- Next.js 기반 웹앱
- TypeScript 사용
- Tailwind CSS 사용
- 사용자 모바일 웹앱과 어드민 웹 페이지를 같은 프로젝트 안에서 관리
- 서버 데이터 관리는 TanStack Query 사용
- 로그인 사용자 상태, 선택한 날짜, 임시 신청 상태는 Zustand 사용 가능
- 폼 검증은 React Hook Form + Zod 사용
- 참가자/신청 데이터는 외부 DB 또는 BaaS와 연동
- 이벤터스 구매자 스프레드시트 데이터는 참가자 DB로 동기화
- 미신청자 명단 다운로드를 위해 CSV 또는 Excel export 기능 제공

## QA 피드백 수신 시

QA_REPORT.md를 받으면:

1. "구체적 개선 지시"를 모두 확인
2. "방향 판단"을 확인
   - "현재 방향 유지" → 기존 코드 수정
   - "완전히 다른 접근" → 디자인 컨셉 근본적 변경
3. 수정 후 SELF_CHECK.md 업데이트
4. "이 정도는 괜찮지 않나?"라고 합리화하지 마라. 피드백을 그대로 반영하라.
