# 하네스 엔지니어링 오케스트레이터

이 프로젝트는 3-Agent 하네스 구조로 동작합니다.
사용자의 한 줄 프롬프트를 받아, Planner → Generator → Evaluator 파이프라인을 자동 실행합니다.

---

## 0. 실행 전 필수 체크 (Think Before Coding)

오케스트레이터 에이전트는 하위 작업을 시작하기 전 반드시 다음을 수행해야 합니다.

1. 명시적인 가정을 기술할 것.
2. 모호한 점이 있다면 임의로 추측하지 말고 사용자에게 즉시 질문할 것.
3. 단계별 실행 계획(Step -> Verify)을 간략히 수립할 것.

---

## 1. 실행 흐름

사용자가 프롬프트를 주면, 아래 순서대로 서브에이전트를 호출합니다.

```
[사용자 프롬프트]
     ↓
          1. SPEC.md 파일 존재 여부 확인
          → 이미 존재하면 Planner 호출 생략 후 ②로 직행
          → 없으면 Planner 서브에이전트 호출하여 SPEC.md 생성
     ↓
          2. Generator 서브에이전트
          → output/index.html 생성 + SELF_CHECK.md 작성
     ↓
          3. Evaluator 서브에이전트
          → QA_REPORT.md 작성
     ↓
          4. 판정 확인
          → 합격: 완료 보고
          → 불합격/조건부: ②로 돌아가 피드백 반영 (최대 3회 반복)
```

---

## 2. 서브에이전트 호출 방법

각 단계에서 사용 가능한 서브에이전트 호출 수단을 사용하여 서브에이전트를 호출합니다.
서브에이전트에게 전달할 프롬프트는 아래 "단계별 실행 지시"를 따릅니다.

중요: 각 서브에이전트는 독립된 컨텍스트에서 실행됩니다.
이것이 "만드는 AI와 평가하는 AI를 분리"하는 핵심입니다.

---

## 3. 단계별 실행 지시

### 단계 1: Planner 호출 및 확인

먼저 현재 디렉토리에 `SPEC.md` 파일이 이미 존재하는지 확인합니다.

- **존재할 경우:** 현재 사용자 요청과 주제가 일치하는 기획서인지 먼저 확인합니다. 일치하면 Planner 단계를 건너뛰고 즉시 [단계 2]로 넘어갑니다.
- **존재하지 않을 경우:** 서브에이전트에게 아래 내용을 전달하여 기획을 지시합니다:

```
agents/planner.md 파일을 읽고, 그 지시를 따라라.
agents/evaluation_criteria.md 파일도 읽고 참고하라.

사용자 요청: [사용자가 준 프롬프트]

결과를 SPEC.md 파일로 저장하라.
```

Planner 서브에이전트가 SPEC.md를 생성하면, 다음 단계로 진행합니다.

### 단계 2: Generator 호출

서브에이전트에게 아래 내용을 전달합니다:

최초 실행 시:

```
agents/generator.md 파일을 읽고, 그 지시를 따라라.
(카파시의 '단순성 우선' 및 '수술적 변경' 원칙을 엄격히 준수할 것)
agents/evaluation_criteria.md 파일도 읽고 참고하라.
SPEC.md 파일을 읽고, 전체 기능을 한 번에 구현하라.

결과를 실제 Next.js 프로젝트 구조로 구현하라.
주요 화면, 컴포넌트, 데이터 흐름, 상태 관리, API 연동 구조를 포함하라.
완료 후 SELF_CHECK.md를 작성하라.
```

피드백 반영 시 (2회차 이상):

```
agents/generator.md 파일을 읽고, 그 지시를 따라라.
agents/evaluation_criteria.md 파일도 읽고 참고하라.
SPEC.md 파일을 읽어라.
output/index.html 파일을 읽어라. 이것이 현재 코드다.
QA_REPORT.md 파일을 읽어라. 이것이 QA 피드백이다.

QA 피드백의 "구체적 개선 지시"를 모두 반영하여 output/index.html을 수정하라.
"방향 판단"이 "완전히 다른 접근 시도"이면 디자인 컨셉 자체를 바꿔라.
완료 후 SELF_CHECK.md를 업데이트하라.
```

### 단계 3: Evaluator 호출

서브에이전트에게 아래 내용을 전달합니다:

```
agents/evaluator.md 파일을 읽고, 그 지시를 따라라.
(단순 작동 여부가 아닌, 목표 중심 검증 원칙을 따를 것)
agents/evaluation_criteria.md 파일을 읽어라. 이것이 채점 기준이다.
SPEC.md 파일을 읽어라. 이것이 설계서다.
output/index.html 파일을 읽어라. 이것이 검수 대상이다.

검수 절차:
1. output/index.html을 분석하라
2. SPEC.md의 기능이 목표대로 구현되었는지 확인하라
3. evaluation_criteria.md에 따라 4개 항목을 채점하라
4. 최종 판정(합격/조건부/불합격)을 내려라
5. 불합격 또는 조건부 시, 구체적 개선 지시를 작성하라

결과를 QA_REPORT.md 파일로 저장하라.
```

### 단계 4: 판정 확인

QA_REPORT.md를 읽고 판정을 확인합니다.

- "합격" → 사용자에게 완료 보고. output/index.html 안내.
- "조건부 합격" 또는 "불합격" → 단계 2로 돌아가 피드백 반영.
- 최대 반복 횟수: 3회. 3회 후에도 불합격이면 현재 상태로 전달하고 이슈를 보고.

## 완료 보고 형식

모든 단계가 끝나면 사용자에게 아래 형식으로 보고합니다:

```
## 하네스 실행 완료

**결과물**: output/index.html
**Planner 설계 기능 수**: X개 (또는 수동 제공됨)
**QA 반복 횟수**: X회
**최종 점수**: 디자인 X/10, 독창성 X/10, 기술 X/10, 기능 X/10 (가중 X.X/10)

**실행 흐름**:
1. Planner: [무슨 기능을 설계했는지 한 줄, 건너뛰었으면 생략 사유 기재]
2. Generator R1: [첫 구현 결과 한 줄]
3. Evaluator R1: [판정 결과 + 핵심 피드백 한 줄]
4. Generator R2: [수정 내용 한 줄] (있는 경우)
5. Evaluator R2: [판정 결과] (있는 경우)
...
```

## 5. 주의사항

- 서브에이전트 호출 시, 반드시 필요한 파일을 읽도록 지시하세요
- Generator와 Evaluator는 반드시 다른 서브에이전트로 호출하여 객관적인 시각을 유지하세요.
- 각 단계 완료 후, 생성된 파일이 실제로 디렉토리에 존재하는지 시스템적으로 확인하세요.
- 동시에 여러 작업이 돌 가능성이 있으면 실행 전 기존 산출물이 현재 작업과 충돌하지 않는지 확인하고, 필요하면 작업별 구분 규칙을 둔다.
- Next dev 서버에서 `./819.js`, `webpack-runtime`, `_not-found` 관련 module not found 오류가 반복되면 `.next` 캐시를 삭제하고, `pages` 폴더 잔존 여부를 확인한 뒤 dev 서버를 재시작한다.
- 3001 포트 오류가 나면 기존 node 프로세스가 남아 있는지 확인하고 종료한 뒤 다시 띄운다.
- QA_REPORT.md를 사람도 읽을 수 있도록, 각 라운드마다 핵심 내용을 요약해주세요

## 6. Next.js 구조 규칙

- `src/app/**/page.tsx`는 라우트 엔트리, 데이터 연결, 상위 조합 역할에 집중한다.
- 큰 UI 블록과 긴 JSX를 `page.tsx`에 계속 누적하지 않는다.
- 공통 UI는 `src/components`로 분리한다.
- 라우트 전용 UI는 `src/app/<route>/_components` 또는 해당 라우트 인접 폴더로 분리한다.
- `page.tsx`가 여러 섹션과 많은 상태, 이벤트 처리, 반복 마크업을 함께 가지기 시작하면 기능 추가 전에 먼저 컴포넌트 분리를 우선 검토한다.
- 기존 `page.tsx`가 이미 큰 경우, 새 기능 추가 시 가능한 범위에서 최소 단위 리팩터링을 함께 수행한다.
- 재사용 여부가 불확실하면 우선 라우트 로컬 컴포넌트로 두고, 반복 사용이 확인되면 공용 컴포넌트로 승격한다.
- 디자인 변경 요청이 아니라면 기존 화면의 톤, 인터랙션, 정보 구조를 유지한다.
